Python Virtual Environments - Complete Guide

python • Knowledge Pillar

Theory & Concepts

What are Virtual Environments?

Virtual environment is an isolated Python environment with its own packages, separate from the system Python.

Real-Life Analogy

Think of separate workspaces:

  • Global Python: Main workshop with shared tools
  • Virtual Environment: Personal workspace with your own tools
  • Different Projects: Different workspaces with different tool versions

Just like you wouldn't mix client A's files with client B's files, you don't mix Project A's packages with Project B's packages!

The Problem Without Virtual Environments

Scenario: You have two projects:

Project A needs: Django 2.2
Project B needs: Django 4.0

Global Python: Can only have ONE Django version!

What happens:

# Install for Project A
pip install Django==2.2
# Project A works! ✅

# Install for Project B
pip install Django==4.0
# Project A breaks! ❌ (Django 2.2 replaced)
# Project B works! ✅

The Solution: Virtual Environments

Global Python: (clean, minimal packages)

Project A Environment:
  └── Django 2.2 ✅
  └── requests 2.25.0

Project B Environment:
  └── Django 4.0 ✅
  └── requests 2.28.0

Both projects work! No conflicts!

Why Virtual Environments are Essential

1. Dependency Isolation

Problem: Different projects need different package versions.

Without venv:

Project 1: needs requests 2.20.0
Project 2: needs requests 2.28.0

Global Python can only have ONE version!
Projects conflict! ❌

With venv:

Project 1 venv: requests 2.20.0 ✅
Project 2 venv: requests 2.28.0 ✅

No conflicts! Both work! ✅

2. Clean Global Python

Without venv: Global Python becomes cluttered

pip list
# 100+ packages from all projects!
# Hard to maintain
# System instability

With venv: Global Python stays clean

pip list  # In global
# Only essential packages

pip list  # In project venv
# Only this project's packages

3. Reproducibility

Team collaboration:

# Developer A creates requirements.txt
pip freeze > requirements.txt

# Developer B recreates exact environment
pip install -r requirements.txt

# Same package versions = No "works on my machine" issues!

4. Testing Different Versions

Test package compatibility:

venv-python-3.9/  # Test with Python 3.9
venv-python-3.10/ # Test with Python 3.10
venv-python-3.11/ # Test with Python 3.11

5. Production Deployment

Deploy exact dependencies:

# Development
pip freeze > requirements.txt

# Production server
pip install -r requirements.txt
# Exact same environment!

Creating Virtual Environments

Using venv (Built-in, Recommended)

Python 3.3+ includes venv module.

Syntax:

python -m venv <environment_name>

Examples:

# Create venv named 'venv'
python -m venv venv

# Create venv named 'env'
python -m venv env

# Create venv named 'myproject_env'
python -m venv myproject_env

What it creates:

venv/
    Scripts/        # Windows (executables)
    bin/            # Mac/Linux (executables)
    Lib/            # Windows (packages)
    lib/            # Mac/Linux (packages)
    Include/        # Header files
    pyvenv.cfg      # Configuration

Alternative: virtualenv

Install:

pip install virtualenv

Create:

virtualenv venv
virtualenv -p python3.9 venv  # Specify Python version

Note: venv (built-in) is sufficient for most use cases.

Activating Virtual Environments

Activation changes your shell to use the virtual environment's Python.

Windows

Command Prompt:

venv\Scripts\activate.bat

PowerShell:

venv\Scripts\Activate.ps1

PowerShell Execution Policy Issue:

# If you get error, run once as admin:
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

# Then activate normally:
venv\Scripts\Activate.ps1

Git Bash (Windows):

source venv/Scripts/activate

Mac / Linux

Bash / Zsh:

source venv/bin/activate

Fish:

source venv/bin/activate.fish

Csh:

source venv/bin/activate.csh

How to Know It's Active

Prompt changes:

# Before activation
C:\Users\Alice\project>

# After activation (venv name in parentheses)
(venv) C:\Users\Alice\project>

Verify:

# Check which Python
where python    # Windows
which python    # Mac/Linux

# Should point to venv directory
# Windows: C:\project\venv\Scripts\python.exe
# Mac/Linux: /project/venv/bin/python

Deactivating Virtual Environments

All Operating Systems

Simple command:

deactivate

What happens:

  • Prompt returns to normal
  • Python returns to system Python
  • pip installs to system again (be careful!)

Example:

(venv) $ deactivate
$ # Back to system Python

Complete Workflow

Project Setup

# 1. Create project directory
mkdir my_project
cd my_project

# 2. Create virtual environment
python -m venv venv

# 3. Activate
# Windows:
venv\Scripts\activate
# Mac/Linux:
source venv/bin/activate

# 4. Install packages
(venv) $ pip install requests pandas

# 5. Save dependencies
(venv) $ pip freeze > requirements.txt

# 6. Work on project...
(venv) $ python main.py

# 7. Deactivate when done
(venv) $ deactivate

Cloning Project

# 1. Clone repository
git clone <repository_url>
cd project

# 2. Create virtual environment
python -m venv venv

# 3. Activate
source venv/bin/activate  # Mac/Linux
venv\Scripts\activate     # Windows

# 4. Install dependencies
(venv) $ pip install -r requirements.txt

# 5. Run project
(venv) $ python main.py

Managing Dependencies

requirements.txt

Create:

(venv) $ pip freeze > requirements.txt

Example requirements.txt:

requests==2.28.0
numpy==1.23.0
pandas==1.4.3
matplotlib==3.5.2

Install from requirements.txt:

(venv) $ pip install -r requirements.txt

Version Specifiers

package==1.0.0    # Exact version
package>=1.0.0    # Minimum version
package<=2.0.0    # Maximum version
package>=1.0,<2.0 # Range
package~=1.4.2    # Compatible release (>=1.4.2, <1.5.0)
package           # Any version (not recommended)

Separate Requirements Files

requirements/base.txt:

Django>=3.2
requests>=2.25.0

requirements/dev.txt:

-r base.txt
pytest>=7.0
black>=22.0

requirements/prod.txt:

-r base.txt
gunicorn>=20.0

Install:

# Development
pip install -r requirements/dev.txt

# Production
pip install -r requirements/prod.txt

Dependency Conflicts

Example Conflict

Package A requires: requests==2.20.0
Package B requires: requests>=2.25.0

Conflict! Cannot satisfy both!

Resolving Conflicts

1. Update packages:

pip install --upgrade package-a
# Hope new version has relaxed requirements

2. Find compatible versions:

# Try different versions
pip install package-a==1.0.0 package-b==2.0.0

3. Use constraint files:

pip install -c constraints.txt package-a package-b

4. Separate virtual environments:

# Project A (uses package-a)
python -m venv venv-a
venv-a/Scripts/activate
pip install package-a

# Project B (uses package-b)
python -m venv venv-b
venv-b/Scripts/activate
pip install package-b

Checking for Conflicts

# Check dependency tree
pip install pipdeptree
pipdeptree

# Check for conflicts
pip check

Best Practices

1. One venv per project

project-a/
    venv/
    src/
    requirements.txt

project-b/
    venv/
    src/
    requirements.txt

2. Never commit venv to version control

.gitignore:

venv/
env/
.venv/
*.pyc
__pycache__/

3. Use descriptive venv names

# Generic (ok)
python -m venv venv

# Descriptive (better for multiple venvs)
python -m venv django-project-env
python -m venv data-analysis-venv

4. Document Python version

runtime.txt:

python-3.11

5. Pin dependencies in production

# Development: flexible
requests>=2.25.0

# Production: exact
requests==2.28.0

6. Regular updates

# Check outdated packages
pip list --outdated

# Update package
pip install --upgrade package-name

# Update all
pip freeze | %{$_.split('==')[0]} | %{pip install --upgrade $_}

Common Issues and Solutions

Issue 1: Activation doesn't work

Windows PowerShell:

# Error: running scripts is disabled
# Solution: Change execution policy
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Issue 2: Wrong Python version

# Specify Python version explicitly
python3.11 -m venv venv

# Or
/usr/bin/python3.9 -m venv venv

Issue 3: venv recreated accidentally

Prevention:

# Don't delete venv folder
# To reset, delete and recreate:
rm -rf venv
python -m venv venv
pip install -r requirements.txt

Issue 4: Forgetting to activate

Indicator:

# No (venv) in prompt
$ pip install package  # Installs globally! ❌

# Activate first
$ source venv/bin/activate
(venv) $ pip install package  # Safe ✅

Issue 5: Large venv size

Solution:

# Don't include data files in venv
# Keep venv only for packages

# Data files elsewhere:
project/
    venv/           # Packages only
    data/           # Data files
    src/            # Source code

VS Code Integration

Select Python Interpreter

1. Ctrl+Shift+P (Cmd+Shift+P on Mac)
2. Type: "Python: Select Interpreter"
3. Choose: ./venv/Scripts/python.exe (or ./venv/bin/python)

Auto-activation

settings.json:

{
    "python.terminal.activateEnvironment": true,
    "python.defaultInterpreterPath": "${workspaceFolder}/venv/bin/python"
}

Terminal Integration

VS Code automatically activates venv when opening integrated terminal!

Advanced: Multiple Python Versions

Using pyenv (Mac/Linux)

# Install pyenv
curl https://pyenv.run | bash

# Install Python versions
pyenv install 3.9.0
pyenv install 3.10.0
pyenv install 3.11.0

# Set project Python version
cd my_project
pyenv local 3.11.0

# Create venv with that version
python -m venv venv

Using py launcher (Windows)

# List installed Python versions
py --list

# Create venv with specific version
py -3.9 -m venv venv39
py -3.11 -m venv venv311
